\section{Introduction}

\subsection{The cell simulation project}
In 2025, I started developing a simulation software for reaction networks called \textit{Cell}~\cite{cell}. The software simulates collisions of circular 2D particles, so-called \textit{Discs}, and reactions thereof. The main output of the simulation is the particle counts over time, as visible in the right-hand side of the screenshot in \cref{fig_cell_screenshot}.

\insertfigure{figures/cell_screenshot.png}{Screenshot of the cell simulation software. The simulation widget on the left shows the simulation state in real time. The plot widget on the right-hand side shows the disc counts over time.}{fig_cell_screenshot}

To simulate a network with \textit{Cell}, a configuration can be created and saved as \texttt{.json} with the GUI. The configuration consists of the following parameters.

\subsection*{Time step}
The time step $\Delta t$ controls the accuracy of the simulation. After each simulation step, every disc is moved by $\Delta \vec{r} = \vec{v}\Delta t$. Smaller $\Delta t$ values make the simulation more accurate but are more CPU intensive.

\subsection*{Most probable speed}
The velocity components of the discs are each sampled from a Gaussian distribution:

\begin{equation*}
    v_x, v_y \sim \mathcal{N}(0, \sigma^2).
\end{equation*}

The standard deviation is calculated using the user-defined most probable speed $v_{\text{mp}}$ and the mass $m$ of the disc:

\begin{equation*}
    \sigma = \frac{v_{\text{mp}}}{m}.
\end{equation*}

\subsubsection*{Disc types}
Every disc in the simulation has a user-defined \textit{disc type}. Each disc type has a name, a mass $m$ and a radius $r$. With the GUI, the user can also associate a color to each disc type which is used to color discs in the simulation and plot widgets. We will call the set of disc types $D$.

\subsubsection*{Reactions}
Two categories of reactions are supported: Unimolecular reactions with time-based probabilities and bimolecular reactions with collision-based probabilities. The simulation supports four kinds of reactions. In the context of this project, only the following two kinds are relevant.

\textit{Transformation reactions} $A \rightarrow B$: The educt changes its disc type such that $m_B = m_A$ and $v_B = v_A$. The reaction probability $p$ is given as a probability for the reaction per second:

\begin{equation*}
    p = 1 - (1-p')^{\frac{1}{\Delta t}},
\end{equation*}

with the simulation time step $\Delta t$ and the success probability $p'$.

\textit{Exchange reactions} $A + B \rightarrow C + D$: For each educt $A$ and $B$, the product most similar in size is chosen and the disc type is changed accordingly: $A \rightarrow X$, $B \rightarrow Y$ with either $X = C$ and $Y = D$ or $X = D$ and $Y = C$. $m_A + m_B = m_C + m_D$ is enforced. For energy conservation, the product velocities are scaled after the reaction:

\begin{equation*}
    v_C = v_X\sqrt{\frac{m_A}{m_X}},\hspace{1cm} v_D = v_Y\sqrt{\frac{m_B}{m_Y}}.
\end{equation*}

The reaction probability $p$ is given as a probability for the reaction after each collision of the educts.\\
The set of all reactions and their probabilities will be called $\Omega$.

\subsubsection*{Simulation size and distribution}
The simulation boundaries are a circle of radius $R$ with a total disc count of $N$. Using the largest disc type radius $r_{\text{max}}$ in the configuration for the edge length, $N$ discs are distributed across a uniform grid. Initial disc counts are specified using a disc type distribution $f$, which assigns relative frequencies to each disc type. Note that depending on $R$, the grid might not fit the requested $N$ discs.

\subsubsection*{The configuration}
With the above notation, each simulation configuration is a vector

\begin{equation*}
    (\Delta t, v_{\text{mp}}, D, \Omega, R, N, f).
\end{equation*}

Using a configuration saved as a \texttt{.json} file, the simulation can be run from the command line by specifying the simulation duration and the storage interval. It outputs a \texttt{.csv} file with the disc counts averaged for each storage interval. If the storage interval is equal to or smaller than $\Delta t$, counts are not averaged and printed after every simulation step. The total simulation duration for all datasets was 60 seconds. This should be enough to see even slow developing prey-predator peaks.

\subsection{Lotka-Volterra dynamics}
\label{subsection_lv_dynamics_intro}
To test the simulation project, a simple prey-predator configuration was created. The configuration consists of the three disc types $D = \{\text{Prey}, \text{Predator}, \text{Resource}\}$ and the following reactions:

\begin{enumerate}
    \item $\text{Prey} + \text{Predator} \rightarrow \text{Predator} + \text{Predator}$ with $p_1$: Predators convert prey to predator.
    \item $\text{Prey} + \text{Resource} \rightarrow \text{Prey} + \text{Prey}$ with $p_2$: Prey converts resource to prey.
    \item $\text{Prey} \rightarrow \text{Resource}$ with $p_3$: Prey dies and transforms back to resource.
    \item $\text{Predator} \rightarrow \text{Resource}$ with $p_4$: Predator dies and transforms back to resource.
    \item $\text{Resource} \rightarrow \text{Prey}$ with $p_5$: Resource transforms to prey, introducing prey to the simulation.
    \item $\text{Resource} \rightarrow \text{Predator}$ with $p_6$: Resource transforms to predators, introducing predators to the simulation.
\end{enumerate}

Initially, the simulation only contains discs of type $\text{Resource}$. With fixed $f$, $\Delta t$, $v_{\text{mp}}$, $N$ and $R$, reaction probabilities were tweaked until the particle counts over time reliably produced the dynamics visible in the example plot in \cref{fig_lv_time_series_example}.

\insertfigure{figures/example_good.png}{Example trajectory of disc type counts using a simple prey-predator configuration over 60s. Prey and predator counts show a distinct periodic characteristic, with prey peaks always followed by predator peaks.}{fig_lv_time_series_example}

The dynamic shown in \cref{fig_lv_time_series_example} is typically referred to as a Lotka-Volterra (LV) dynamic (see \cref{subsection_lv_dynamics_related}). The goal of this project is to find the region of the six dimensional probability space for some fixed $f$, $\Delta t$, $v_{\text{mp}}$, $N$ and $R$ that is likely to exhibit LV dynamics. To this end, two binary classifiers were trained on a dataset obtained from the simulation: One using traditional deep learning methods with a single model, and a second using averaged weights from multiple models trained on subsets of the dataset using FL methods. The performance of these models is then compared using various scoring metrics. Additionally, the impact of IID and non-IID dataset splitting and the number of clients on the FL model performances is evaluated in comparison with the baseline model. 

\section{Related work}
\subsection{Lotka-Volterra equations}
\label{subsection_lv_dynamics_related}
The dynamic shown in \cref{fig_lv_time_series_example} is very similar to dynamics exhibited by solutions of the well-known Lotka-Volterra equations~\cite{wikipedia_lv_equations}:

\begin{align*}
    \frac{dx}{dt} = \alpha x - \beta xy,\\
    \frac{dy}{dt} = -\gamma y + \delta xy.
\end{align*}

Here, $x$ and $y$ represent the prey and predator populations and the parameters $\alpha$, $\beta$, $\gamma$ and $\delta$ have similar roles to the probabilities $p_i$ of the above configuration. The solutions of these equations are however deterministic, assume a well-mixed environment and do not model a third reaction species like the resource in the above model with disc types. As long as the initial prey and predator counts and the parameters are all strictly positive, solutions of the Lotka-Volterra equations always show phase-shifted, periodic prey-predator peaks. In the more general reaction model with disc types, only a small region of the parameter space shows this dynamic. And since this is a custom developed model, it is unlikely that similar systems have been studied like this before.

\subsection{Impact of dataset splitting on Federated Averaging}
The impact of increasing number of clients and dataset splitting on the performance of models obtained using Federated Averaging is already discussed in the original Federated Averaging paper~\cite{McMahan2017}. 

\section{Methodology}
The project consists of three parts. First, a LV-classifier was developed to label data. It was designed to be simple and fast enough to quickly label thousands of CSV-files with disc count trajectories. Second, a simple dataset where only $p_1$ and $p_2$ (see \cref{subsection_lv_dynamics_intro}) were varied was created. The other four probabilities were fixed at values that are known to exhibit LV-dynamics. The resulting 2D probability space should give a first impression of whether the region of interest is closed or not. A simple model was trained on this smaller dataset to judge if the MLP approach is sufficiently accurate. Third, a larger dataset spanning the entire 6D probability space was created. Two different model types were trained with this data:

\begin{itemize}
    \item A single baseline model using the entire dataset for training
    \item $2N$ models with averaged weights obtained from $N$ client models by splitting the dataset
    \begin{itemize}
        \item randomly
        \item into cohesive 6D regions
    \end{itemize}
\end{itemize}

\subsection{Model topology}
All models were Multi Layer Perceptron (MLP) classifiers. With enough hidden units, MLPs can approximate complex multi-dimensional decision boundaries, needing relatively few parameters for the task in this project. In a FL setting, they require relatively low communication cost and train quickly on local clients. A Support Vector Machine (SVM) or logistic regression are possible candidates in case of poor MLP model performance. Unless stated otherwise, any dataset for training was split 80/20 into train and test data.

All MLPs used have an input node for each probability (up to six), two hidden layers with 64 nodes each, and a single output node. The chosen activation function is ReLU ($f(x) = \max(0, x)$), which is a sensible default for binary classification. The learning rate and weight decay remained at the default values of $0.001$ each. To avoid varying internal representations of the data, the training data was scaled to zero mean and unit variance. The same scaler was used to scale the test data as well and was only fitted using the training data. 

\subsection{Datasets}
Datasets were created by calling the CLI of the cell project from python scripts using \texttt{concurrent.futures.ProcessPoolExecutor} on Linux with four workers. The small 2D datasets where only $p_1$ and $p_2$ were varied was created by taking a configuration that was known to reliably exhibit LV dynamics according to the developed classifier and creating a $101 \times 101$ grid with $p_1, p_2 \in \mathbb{Z} \cap [0, 100]$.

The larger 6D datasets used Latin hypercube sampling with $N = 20\,000$ across the whole 6D probability space. Latin hypercube sampling divides each dimension into $N$ equally sized intervals and generates $N$ samples such that each interval occurs exactly once in every dimension. One sample is then drawn uniformly from each interval. This ensures the samples lie roughly uniformly in 6D space, avoiding local clusters that could occur if, for example, random sampling was used instead. 

\subsection{Federated Learning}
Since the dataset was already fully available, federated learning was simulated by splitting the training split of the dataset used for training the baseline model into several smaller ones. The test dataset was not visible to any clients and only used to evaluate the final model. Additionally, heatmaps on the $p_1 \times p_2$ plane were compared for each model. The same \texttt{StandardScaler} from the baseline model was used by all models involved to avoid a varying internal representation of the data. Training utilizes the Flower framework with federated averaging, following these steps:

\begin{enumerate}
    \item The server is initialized with the global model weights $w_t$
    \item These weights are sent to each participating client 
    \item Each client trains locally for five epochs 
    \item Each client then returns the updated weights to the server 
    \item The server computes a weighted average of the weights
    \item Repeat from step 2. ten times to have 50 epochs in total
\end{enumerate}

The dataset was split either via a stratified shuffled group split or uniformly along the $p_1$ axis to examine the impact of identically and independently distributed (IID) data versus non-IID data. Stratification ensures that all clients get roughly the same number of positive and negative labels. Since each 6D input vector corresponds to five simulation runs, a client should always see all five runs to be able to learn a meaningful probability. This is why all five runs were grouped together and the groups were then distributed across the clients.

\subsection{Scoring methods}
Performance of the models was judged by comparing 2D heatmaps of the decision boundary with simulated data and by calculating the ROC AUC, log loss and the brier score. As a shorthand, a \enquote{score} in the context of this project is a tuple of the form $(\text{ROC AUC}, \text{log loss}, \text{brier score})$. 

\subsubsection{ROC AUC}
The receiver operating characteristic (ROC) is calculated by varying the positive label threshold for the predicted probability of a binary classifier and plotting the true positive rate against the false positive rate. A perfect classifier goes through the point $(0, 1)$, which means the model perfectly separates all true positives from false positives. Since the curve is monotonically increasing (adding additional false positives will never decrease the true positive rate), the area under the curve (AUC) for the perfect model is $1$ (area of unit square). A random classifier would follow the identity line and have an AUC of $0.5$. The closer a model is to this line, the worse it performs. An AUC of $< 0.5$ is technically even worse, but then the model's output could just be inverted to obtain an AUC of $> 0.5$ again. An example plot is shown in \cref{fig_roc_auc}.

\insertfigure{figures/roc_auc.png}{Example diagram for the ROC curve from~\cite{wikipedia_roc}. The ROC curve is obtained by varying the decision threshold for labeling and plotting the true positive rate against the false positive rate. The area under the curve (AUC) is then used as the metric.}{fig_roc_auc}

\subsubsection{Brier score}
The \textit{brier score} is the mean squared error:

\begin{equation*}
    \text{Brier score} = \frac{1}{N}\sum_{i=1}^N (p_i - y_i)^2,
\end{equation*}

where $p_i \in [0, 1]$ are the predicted probabilities and $y_i \in \{0, 1\}$ are the true labels. The brier score measures the overall accuracy of the model, where a score of $0$ is best and $1$ is worst.

\subsubsection{Log loss}
The binary cross entropy or \textit{log loss} is calculated via 

\begin{equation*}
    \text{Log loss} = -\frac{1}{N}\sum_{i=1}^N\left(y_i\log(p_i)+(1-y_i)\log(1-p_i)\right).
\end{equation*}

For $y_i = 0$, the summand contributes $\log(1-p_i)$, which grows large for $p_i \rightarrow 1$. For $y_i = 1$, the summand contributes $\log(p_i)$, which grows large for $p_i \rightarrow 0$. This means that the log loss particularly penalizes confident wrong predictions. The best score would be $0$ and the worst can be arbitrarily large.

\section{Results}
\subsection{Developing a LV-classifier}
The definition of "LV-like dynamics" in the context of this project is mostly qualitative and refers to the following characteristics:

\begin{itemize}
    \item Prominent prey-predator peaks that appear in pairs
    \item The peaks are phase-shifted with the predator peaks lagging behind the prey peaks
\end{itemize}

\Cref{fig_lv_good_bad} shows the classifier in action. The classifier smoothes the curves using \path{scipy.signal.savgol_filter} and mostly relies on the \texttt{prominence} parameter of \path{scipy.signal.find_peaks}, which measures how much the peak stands out from its surrounding~\cite{prominence}.

\insertfigure{figures/lv_good_bad.jpg}{LV classification of two prey-predator count trajectories. The smoothed curves were drawn on top of the original count trajectories. Trajectories are labeled as yes or no depending on how many peaks are found and if predator peaks lag behind prey peaks. Peaks are found using \texttt{scipy.signal.find\_peaks} and marked with dots.}{fig_lv_good_bad}

To find the optimal parameters for the classifier, a 2D dataset was created by varying $p_1$ and $p_2$ in steps of 1\% with fixed other probabilities. Each simulation was run five times. The prominence was varied by viewing samples at the decision boundary and seeing if the LV dynamic was pronounced enough, i.e. the simulation showed clearly visible prey and predator population peaks. A comparison of the results using two different prominences is shown in \cref{fig_lv_prom_comp}.

\insertfigure{figures/simple_prom_comp.jpg}{Probabilities assigned by the LV classifier to the 2D dataset for different values of the \texttt{prominence} parameter. A prominence of 50 for both prey and predator peaks yielded many positive results that showed only weak LV dynamics after reviewing. A prominence of 150 for prey and 120 for predator peaks showed a much smaller positive region with very few outliers.}{fig_lv_prom_comp}

\subsection{Training a model on a 2D subspace}
\label{subsection_simple_model}
Using the 2D dataset where only $p_1$ and $p_2$ were varied, a simple MLP classifier with two input nodes was trained. Since each unique combination or \textit{group} was simulated five times, using group shuffling for the train and test datasets was necessary avoid unique combinations to appear in both. The decision boundary of this model and the original LV classification results are compared in \cref{fig_simple_model_comp}.

\insertfigure{figures/simple_comp.jpg}{Comparison of empirically estimated and model-predicted probability landscapes on a fixed $100 \times 100$ grid.}{fig_simple_model_comp}

The score of this model is $(0.984, 0.064, 0.019)$. The highest predicted probability is $0.94$.

\subsection{Training a model on the full 6D space}
\label{subsection_baseline_model}
With the available CPUs for this project a total of about 100,000 simulations could be run in a day. For this dataset, 20,000 unique parameter vectors were run five times each. The dataset was created using Latin hypercube sampling. For $p_i \in [0, 1]$ this resulted in less than five positive results. To obtain a more targeted dataset, the viable range of $p_1 \in [0.1, 1]$ and $p_2 \in [0, 0.2]$ from \cref{fig_simple_model_comp} was used instead. $p_3$ through $p_6$ were fixed to $\pm 0.1$ (clamped to $[0, 1]$) within the value used for the runs that resulted in \cref{fig_simple_model_comp}. The targeted dataset was created using Latin hypercube sampling with subsequent scaling to the bounds. This dataset resulted in a total of about 5,000 positive hits with about 2,200 ($\approx 10\%$) unique probability vectors. Again, a MLP model, this time with six instead of two input nodes, was trained on the full dataset and also tested on the dataset from \cref{fig_simple_model_comp}. The results are shown in \cref{fig_full_model_comp_1}.

\insertfigure{figures/full_comp.jpg}{Comparison of empirically estimated and model-predicted probability landscapes on a fixed $100 \times 100$ grid.}{fig_full_model_comp_1}

The model extrapolated incorrectly in the region of $p_1 \in [0, 0.1]$. Subsequently, a third and final dataset with $p_1 \in [0, 1]$ was created and used to train another MLP classifier. Its performance on the $p_1 \times p_2$ plane is shown in \cref{fig_full_model_comp_2}.

\insertfigure{figures/full_comp_2.jpg}{Predicted label probabilities of the model trained on the dataset with $p_1 \in [0, 1]$ instead of $p_1 \in [0.1, 1]$.}{fig_full_model_comp_2}

The incomplete model from \cref{fig_full_model_comp_1} had the score $(0.971, 0.088, 0.027)$, whereas the corrected one had $(0.963, 0.090, 0.027)$. The highest predicted probability of the full model was $0.92$. 

\subsection{FL models}
The scores as a function of the number of clients are shown in \cref{fig_scores_comp}.

\insertfigure{figures/scores_comp.jpg}{ROC AUC, brier score and log loss as a function of the number of clients for models trained on either the IID (random) or non-IID (cohesive) dataset splits.}{fig_scores_comp}

The predicted probabilities of the FL models using IID and non-IID dataset splits for $N=2$ and $N=64$ are shown in \cref{fig_fl_heatmap_n2_iid,fig_fl_heatmap_n64_iid,fig_fl_heatmap_n2_non_iid,fig_fl_heatmap_n64_non_iid}. 

Even for $N = 2$, the maximum predicted probability on the $p_1 \times p_2$ plane was $0.73$ for the model trained on IID data, and $0.64$ for the one trained on non-IID data. This is significantly lower compared to $0.92$ for the baseline model (see \cref{subsection_baseline_model}).

\section{Discussion}
\subsection{The simple 2D model}
According to \cref{fig_simple_model_comp}, the simple MLP model was able to approximate the positive region quite well. Together with the good accuracy scores of this model, this indicates the model can correctly predict how likely any given combination of $(p_1, p_2)$ will result in LV dynamics. Since MLPs seem to be a good fit for this problem, SVMs, logistic regression or other approaches were not attempted.

\subsection{The full 6D baseline model}
The small number of positive results in the first 6D dataset spanning the entire probability space indicate a very small surface area of the interesting region in 6D space. The second training dataset only contained values for $p_1 \in [0.1, 1]$ and the model didn't learn the correct distribution for $p_1 \in [0, 0.1]$ (see \cref{fig_full_model_comp_1}). Since it wasn't included in the training dataset and nearby empirical values were positive, the model incorrectly extrapolated a high label frequency into this region. 

The scores for the incorrectly extrapolating model were even slightly better than the one correctly trained on the missing region. This exemplifies how scores can only be meaningfully interpreted if the used testing dataset is appropriate.

\subsection{FL models}
According to \cref{fig_scores_comp}, the scores degrade with increasing number of clients. The models trained on the IID data perform consistently better than the ones trained on non-IID data. This is in line with expectations: A sparser local dataset will negatively impact prediction performance of locally trained models, and since the global model is obtained by averaging local model updates, it reflects this characteristic.

As expected, the performance of the FL models approached that of the baseline model when fewer clients were used.  Interestingly, even for small $N$ such as $N=2$ the confidence of the model, i.e. its highest predicted probability, is quite low for both dataset splits. This is most likely due to the MLP trying to minimize the log loss in regions with missing data, which is sensitive to confident wrong predictions. 

\section{Conclusion}
The MLP model proved to be successful in predicting the likelihood of LV dynamics as defined in this project in 6D probability space. Since visualizing the entire 6D space is challenging, it cannot be concluded if the baseline model, and subsequently the FL models, fully learned the decision boundary. Other regions may exist. However, as \cref{fig_simple_model_comp} showed, the model is capable of learning a given region in 6D subspace, indicating that if all regions are present in the training dataset, they were likely learned as well.

The heatmaps in \cref{fig_fl_heatmap_n2_iid,fig_fl_heatmap_n2_non_iid,fig_fl_heatmap_n64_iid,fig_fl_heatmap_n64_non_iid} indicate that the models obtained using federated averaging reproduced the overall shape of the decision boundary, but predicted substantially lower probabilities throughout the positive region, even for just $N=2$ clients. This is an expected result of sparse client datasets.

Future work could include full visualization of the 6D space to identify the whole decision boundary. It would be interesting to see how the other simulation parameters (see \cref{subsection_lv_dynamics_intro}) influence the decision boundary and how that can be linked to actual real life systems or the Lotka-Volterra equations.
